### 1편 — HTML 스크래핑과 JSON API 직접 호출의 비교

- **분류**: 기초
- **사례 코드**: `sites/iris/crawler.py` 전체. 특히 `LIST_API_URL`(L52), `_fetch_list_page`(L92-107), `_row_from_api`(L110-133).
- **대비 사례**: 다른 사이트들의 HTML 파싱 흐름(`sites/keit/crawler.py`, `sites/smtech/crawler.py`)을 1~2줄 인용으로 대조.
- **AI가 한 일**: 캡처한 `POST /contents/retrieveBsnsAncmBtinSituList.do` 요청을 보고 파라미터(`ancmPrg`, `pageIndex`, `recordCountPerPage`)의 의미 추정 + JSON 응답 키 매핑(`ancmId`, `ancmTl`, `rcveEndDe` 등) + 행 1개를 한국어 키 딕셔너리로 정규화하는 코드 생성.
- **사람이 한 일**:
  - "이 사이트는 HTML 파싱이 답이 아니다"를 식별. 브라우저 Network 탭에서 화면 렌더링과 무관한 JSON 응답이 더 풍부함을 관찰.
  - 응답이 JSON이라는 사실이 곧 "공식 API"를 의미하지 않는다는 점을 인지하고 적용 범위를 한정.
- **결론(사람에게 남는 일 한 줄)**: 화면 뒤편에서 더 안정적이고 단순한 인터페이스가 흐르고 있는지를 식별하는 **인터페이스 감별 능력**.
- **본문 30 % 코드 자리**: `_fetch_list_page` + `_row_from_api` 12~15줄.

### 2편 — requests.Session 과 세션 쿠키

- **분류**: 기초
- **사례 코드**: `sites/iris/crawler.py:72-77` `new_session()`.

  ```python
  def new_session():
      s = requests.Session()
      s.headers.update({"User-Agent": ..., "Accept-Language": "ko,en;q=0.8"})
      # 세션 쿠키(JSESSIONID) 확보 — AJAX가 막힐 경우 대비
      s.get(LIST_VIEW_URL, timeout=20)
      return s
  ```

  + 공통 헤더 정의 `core/http.py:7-12`.
- **AI가 한 일**: `requests.Session` 패턴, `User-Agent` 설정, `Referer` 헤더 부여.
- **사람이 한 일**:
  - 첫 AJAX 호출이 빈 배열을 반환했을 때 "왜?"를 끝까지 추적. 두 가지 가설(① 파라미터 누락, ② 세션 쿠키 부재) 중 후자를 우선 시험.
  - 진단 절차: 브라우저에서는 되는데 코드에서는 안 됨 → 두 요청의 헤더·쿠키 비교 → JSESSIONID 발견 → 화면 진입 GET을 한 번 추가.
- **결론**: 같은 HTTP인데 결과가 달라질 때 **두 요청의 차이를 한 줄씩 비교하는 절차**를 가지고 있는지가 진짜 능력이다. AI는 그 절차의 결과(코드 한 줄)는 잘 쓰지만, 그 절차를 시작할 시점은 사람이 결정한다.
- **본문 30 % 코드 자리**: `new_session` + AJAX 호출의 `Referer` 헤더 부분.

### 3편 — AJAX 엔드포인트의 비공식 분석 절차

- **분류**: 응용
- **사례 코드**: IRIS·SMTECH·NIPA·KEIT 4사이트 모두 AJAX 기반. 4사이트의 엔드포인트 표 1개로 정리(POST URL, 파라미터, 응답 구조 키 3개씩).
- **AI가 한 일**: 사용자가 캡처해 던진 cURL 한 줄을 받아 Python `requests` 호출로 변환 + 응답 키 매핑 + 페이지네이션 루프.
- **사람이 한 일**:
  - F12 → Network → XHR → 페이지 동작 재현 → 요청 캡처. 이 5단계는 사람이 한다.
  - **윤리·정책 판단**: 비공식 호출의 적절한 한도 결정. rate-limit, `robots.txt`, 사이트 안내 페이지 확인 → "공고 조회 수준이면 일반 사용자 행위와 다르지 않다"는 판단으로 진행. 첨부 다운로드의 빈도는 일 1회로 제한.
  - **안정성 위험**: 사이트 개편 시 즉시 깨질 수 있음을 인지하고, 어댑터 단위로 격리(`sites/<name>/`)해 한 사이트가 깨져도 다른 사이트는 계속 돈다는 구조를 선택.
- **결론**: 비공식 인터페이스를 쓸 때 사람은 **어디까지가 적절한 사용인가의 한도**를 정한다. AI는 한도를 묻지 않는다.
- **본문 30 % 코드 자리**: IRIS 1사이트의 page1 응답 JSON 발췌(`paginationInfo`, `listBsnsAncmBtinSitu` 5개 키만).

### 4편 — 한국어 ZIP 파일 인코딩(CP949) 문제의 정확한 원인

- **분류**: 응용
- **사례 코드**: `core/archive.py:15-26` `_decode_member_name`.

  ```python
  if info.flag_bits & 0x800:   # UTF-8 플래그가 있으면 그대로
      return name
  try:
      return name.encode("cp437").decode("cp949")   # 없으면 CP949로 재해석
  except (UnicodeDecodeError, UnicodeEncodeError):
      return name
  ```

- **AI가 한 일**: `cp437 → cp949` 재해석 코드 + zip slip 방어(`out_path.resolve()` 검사).
- **사람이 한 일**:
  - "왜 일부 파일은 깨지고 일부는 멀쩡한가" 추적. `zipfile.ZipInfo.flag_bits & 0x800` 비트(ZIP 명세 4.4.4의 Language encoding flag) 의미를 직접 확인하고 코드에 반영하라고 지시.
  - 단순 "한국어니까 CP949"가 아니라 ZIP 명세에 정의된 분기를 따른다는 정확성 요구. AI 초안은 무조건 CP949로 디코딩하려 했고 이는 UTF-8 ZIP에서 역으로 깨졌다.
- **결론**: 표준이 정의한 분기가 있을 때 그 분기를 따르도록 코드를 강제하는 일은 사람이 한다. AI의 첫 답은 "대표적 케이스 한 가지"에 머무르기 쉽다.
- **본문 30 % 코드 자리**: 위 6줄 + zip slip 방어 2줄.

### 5편 — 증분 수집 전략: 워터마크 vs 상태 기반 스킵

- **분류**: 응용
- **사례 코드**:
  - `core/incremental.py` — 등록일 기준 워터마크, list 단계 early-stop (`should_stop_list`, L72-90).
  - `core/catalog.py:44-88` — item_id 기준 upsert, "접수중"의 자동 만료.
  - state 키 형식: `site:item_id` (`SCOPE.md:46-50`).
- **AI가 한 일**: 두 메커니즘 모두 코드 작성.
- **사람이 한 일**:
  - "왜 두 단계 모두 필요한가" 설계 결정. list 단계는 페이지 부담을 줄이고, download 단계는 정확한 중복 방지를 한다. 한 단계만으로는 부족한 이유를 명문화.
  - **미래 날짜 함정 발견**: SMTECH 예고공고에 2102년 등록일이 등장 → 그대로 max를 취하면 그 이후 항목이 영원히 차단됨. 워터마크 함수에 "오늘 이전" 필터 추가하도록 요구(`incremental.py:37-46`).
  - **`--full-list` 보정 모드**: 등록일 기반 early-stop이 누락한 항목 보정용. 주 1회 강제 전체 순회 옵션을 명시.
- **결론**: 자동화의 효율과 정합성은 서로를 배신할 수 있다. 사람은 **정합성 보정 장치를 같이 설계한다**.
- **본문 30 % 코드 자리**: `should_stop_list` 8줄 + 미래 날짜 필터 4줄.

### 6편 — 임시 데이터와 영구 데이터의 분리 설계

- **분류**: 실전
- **사례 코드**: `core/paths.py:8-13`.

  ```python
  DOWNLOADS = OUTPUT / "_downloads"   # 첨부 폴더 (주기적으로 비울 수 있음)
  STATE_DIR = OUTPUT / "_state"       # 중복방지·이력 (절대 자동 삭제 X)
  ```

  + `_migrate_legacy`(L25-61): 이전 위치에서 자동 이동.
- **AI가 한 일**: 마이그레이션 로직.
- **사람이 한 일**:
  - 운영 요구를 한 문장으로 명시: **"다운로드 폴더 안 공고 폴더는 사용자가 임의로 비워도 다음 실행이 망가지지 않아야 한다."**
  - 이전 구조(`_state.json`이 `_downloads/` 안)에서는 다운로드 폴더를 비울 때 state까지 같이 날아가는 사고가 가능했다. 분리 결정 + 자동 마이그레이션 요청.
- **결론**: 데이터마다 **수명과 관리자**가 다르다. 시스템 설계는 그 차이를 폴더 경계로 드러낸다. 이 결정은 사람이 한다 — AI는 묻지 않으면 한 폴더에 다 넣는다.
- **본문 30 % 코드 자리**: `paths.py` 상단 정의 6줄 + 마이그레이션 핵심 5줄.

### 7편 — 같은 데이터가 여러 출처에 있을 때 단일 진실원(SSOT) 결정

- **분류**: 실전
- **사례 코드**:
  - `sites/keit/crawler.py` 의 `INCLUDE_BIZ_BOARD = False` 토글.
  - `SCOPE.md` 의 수집 범위 매트릭스.
  - state 키 prefix(`iris:`, `keit:`) 분리로 충돌 방지.
- **AI가 한 일**: 4개 사이트 어댑터를 모두 만들어 "다 수집" 형태로 동작시킴.
- **사람이 한 일**:
  - **관찰**: 정부 R&D 공고 본문은 KEIT/SMTECH/NIPA/IRIS에 동시에 올라온다. 누적 다운로드의 상당 부분이 같은 공고의 4중 복제.
  - **결정**: IRIS를 R&D 공고의 SSOT로 지정. 다른 사이트는 R&D 본공고 항목을 끄고 그 사이트 고유 항목(인터넷공시·수요조사 등)만 남김.
  - **트레이드오프 명시**: 전수 수집의 안전망(한 사이트 장애 시 다른 사이트로 복구)을 잃는 대신, 다운로드 비용·중복 분석 비용을 4분의 1로 절감.
- **결론**: AI는 "어디서 수집할 것인가"는 묻지 않고 "각각 어떻게 수집할 것인가"만 푼다. **데이터의 출처가 여러 개라는 사실은 사람이 발견하고, 어느 출처를 진실로 삼을지도 사람이 결정한다.**
- **본문 30 % 코드 자리**: `INCLUDE_BIZ_BOARD` 토글 부분 + `SCOPE.md` 표 1개.

### 8편 — 더블클릭 한 번을 위한 설계와, 사람의 수고를 또 줄이는 시도

- **분류**: 실전
- **사례 코드**:
  - `run.bat` 전체 45줄, `run_all.py:21-60` argparse 인터페이스.
  - 통합 결과 표면: `output/_state/_ledger.xlsx` 한 파일 (A 시스템), Streamlit `app_db.py` 4탭 단일 URL (B 시스템).
  - Claude Code `settings.local.json` 의 권한 자동 허용 항목 (yes 클릭 줄이기).
- **AI가 한 일**: CLI 인자·argparse·사이트 선택 옵션·대시보드 탭 코드.
- **사람이 한 일**:
  - **시나리오 명문화**: "매일 아침 출근 후 `run.bat` 더블클릭. 결과 확인은 `_ledger.xlsx` 또는 `localhost:8501` 한 곳."
  - 결과 확인 표면을 1개로 압축 — 사이트별 산출물이 따로 있어도 사용자가 보는 정식 산출물은 통합 파일 하나뿐.
  - **사람의 수고 자체를 자동화 대상으로 보기**: Claude Code의 `settings.local.json` 으로 반복 yes 클릭 항목을 미리 허용 / 표본 검증 결과가 OK일 때 다음 단계 자동 진행 / 에러 시에만 사람을 부르도록 출력 단순화.
  - **"한 번의 클릭"을 시스템 구조로 보장하기**: 사용자가 보는 한 번의 실행 안에 모든 결정 지점을 사전에 정해두는 일이 사람의 일. 일상 운영에서 사람이 다시 결정하지 않도록 하는 것이 메타 자동화의 핵심.
- **결론**: 비개발자가 만든 도구의 진짜 사용성은 **명령줄 너머의 한 번의 클릭**과 **결과를 확인하는 한 개의 파일**로 압축된다. 그리고 그 한 번의 클릭 안에 사람의 결정이 압축되어 있다.
- **본문 30 % 코드 자리**: `run.bat` 발췌 10줄 + ledger.xlsx 컬럼 표 + `settings.local.json` 의 권한 항목 발췌.

### 9편 — (종결) 사람에게 남는 일 + 여러 프로젝트의 관리 패턴

- **분류**: 종합
- **구조**:
  1. **1~8편 "사람에게 남는 일" 한 줄 결론을 표로 모음.**
  2. **패턴 분류**: ① 인터페이스/포맷 감별, ② 정합성·운영 정책 수립, ③ SSOT·수명·경계 결정, ④ 사용자 시나리오 명문화, ⑤ 비공식 사용의 한도 결정, ⑥ 메타 자동화(사람의 수고를 줄이는 시도).
  3. 각 패턴의 공통점: **모두 "무엇을 만들지 않을 것인가"의 결정**이다. AI는 "어떻게 만들지"는 다 푼다.
  4. **다중 프로젝트 관리 패턴**: 한 사람이 N개의 미니 자동화를 동시에 운영하는 방법. 다루는 도구:
     - 전역 `~/.claude/CLAUDE.md` (사용자 프로파일·연속성 규칙)
     - 프로젝트별 `CLAUDE.md` + `NOTES.md` (현재 상태·다음 할 일·막힌 부분)
     - `memory/` 디렉터리 (세션 간 정착되는 사실·피드백·참조)
     - 블로그 자체가 그 메타 관리의 또 다른 층 — 결정의 기록을 외부 공개로 강제하는 장치
  5. 잠정 결론: 비개발자가 코딩 에이전트와 함께 일할 때 갖춰야 하는 능력은 **요구 정의력 + 검증력 + 한도 설정력 + 메타 자동화 설계력**의 네 항으로 정리된다.
  6. 다음 시리즈 예고: 이 도구들을 6개월 운영하면서 깨진 부분과, 사람의 개입이 어디서 또 필요했는지의 후속 기록.
- **본문 분량**: 본 회차는 1~8편 발췌 + 종합 표 + 다중 프로젝트 관리 사례가 본체. 코드 인용 5 % 이하.
