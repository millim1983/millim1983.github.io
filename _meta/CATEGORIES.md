# 카테고리 정의서

이 블로그의 글은 모두 아래 9개 카테고리 중 하나(또는 둘)에 속한다. 새 글을 쓸 때 front matter에 다음을 적는다.

```yaml
categories: [<슬러그>]
```

여러 카테고리에 동시에 속하는 글은 거의 없다. 일단 한 곳에 두고 필요하면 태그(`tags:`)로 보완한다.

---

## 1. 내 생각 (`thoughts`)

**무엇이 들어가나**: Claude Code·AI 시대 사용자의 자리에 대한 짧은 견해. "코드"가 아니라 "도구"라는 관점에서, 이 변화가 일하는 방식·교육·일상에 미치는 영향에 대한 단상.

**톤**: 일기보다는 짧고 정리된 의견. 큰 주장이 아니라 한두 문단의 관찰.

**예시 글감**:
- Claude Code를 처음 1년 써보고 든 생각
- "AI가 다 한 거 아니냐"는 평가에 대한 한 줄 답
- 도구가 한 번 자리를 옮긴 역사 — 엑셀·카메라·내비게이션·번역

**분량**: 600~1500자.

**발행 풀**: 주로 `light/` 풀. 길어지면 `deep/`.

## 2. AI와 개발하기 (`building-with-ai`)

**무엇이 들어가나**: 실제로 만든 시스템 하나를 사례로 잡고, "도구가 한 일"과 "사용자가 결정한 일"을 분리해 기록하는 시리즈. 본 블로그의 메인 줄기 중 하나.

**톤**: 진지·기록. 코드 인용 30~40 %, 의사결정·일반화 60 % 이상 (PERSONA.md § 1·4 참고).

**현재 진행 시리즈**: 9편짜리 첫 시리즈. 회차 매핑은 `SERIES_OUTLINE.md` 참고.

**예시 글감**:
- HTML·AJAX·OpenAPI 세 층위의 진짜 차이 (정부 공고 + 나라장터)
- 한글 데이터의 흔한 함정들 (CP949·HWP·콘솔 인코딩)
- 같은 정보가 여러 출처에 있을 때의 단일 진실원(SSOT) 결정

**분량**: 2500~5000자. 한 시스템·한 결정에 집중.

**발행 풀**: `deep/`.

---

## 3. 소프트웨어 디자인·아키텍처 (`design-architecture`)

**무엇이 들어가나**: **큰 그림** 구조에 대한 글. 모듈 간 관계, 책임 분리, 데이터 흐름, 의존성 관리, 디자인 패턴(Strategy/Factory/Adapter/Observer 등), SOLID, 레이어드·플러그인 아키텍처, 헥사고날, 이벤트 주도, 트랜잭션 경계 등.

**톤**: **바이블 톤**. CS 이론 + 실전 예시. AI가 짠 코드를 리뷰어 입장에서 큰 그림으로 읽고 평가하는 데 필요한 역량을 길러주는 글들. 한 글은 한 개념의 정의 + Claude Code가 만든 실제 코드에 그 개념이 어떻게 나타났는지의 매핑으로 구성.

**예시 글감**:
- 플러그인 어댑터 패턴 — `biz-announcement-monitor`의 사이트별 `sites/<name>/` 폴더가 왜 이 구조인가
- 단일 책임 원칙(SRP)을 Claude가 자발적으로 어긴 사례와 사람의 교정
- 의존성 역전(DIP) — `core/`가 `sites/`를 모르는 이유

**분량**: 3000~6000자. 개념 정의(바이블) + 코드 인용(실전).

**발행 풀**: `deep/`.

---

## 4. 클린 코드 (`clean-code`)

**무엇이 들어가나**: **작은 그림** — 함수·한 줄 단위의 가독성, 명명(naming), 리팩토링, 시간/공간 복잡도, 코드 스멜, 작은 단위 관용구. Robert C. Martin의 *Clean Code*, Martin Fowler의 *Refactoring*, *Effective C++/Java* 시리즈 풍의 처방형 글.

**디자인·아키텍처와 차이**: 디자인은 "모듈 간", 클린 코드는 "한 함수 안". 둘 다 다루는 글은 카테고리 한 곳에 두되 본문에서 자연스럽게 연결.

**예시 글감**:
- Claude가 자동 생성한 변수명 `temp_data`를 검토할 때 보는 것
- 매개변수 3개 넘는 함수를 어떻게 다시 짤지
- 일찍 반환(early return)으로 if 중첩 풀기

**분량**: 1500~3000자. 한 글 한 처방.

**발행 풀**: `light/` 또는 `deep/`.

---

## 5. 알고리즘 (`algorithms`)

**무엇이 들어가나**: 두 갈래.
- **고전 알고리즘과 AI 짠 코드의 매핑**: Claude가 자동으로 쓴 코드가 알고 보니 고전 알고리즘 한 개와 같았다는 발견들. 예) `core/incremental.py`의 워터마크 비교 = 단조 함수의 이분 탐색 사촌.
- **유명 알고리즘 학습 노트**: 정렬·탐색·그래프(BFS/DFS/다익스트라)·동적 계획법·해싱·문자열(KMP·suffix array)·근사·확률 알고리즘 등을 하나씩 정리. 출처는 정통 교재(CLRS, Sedgewick, Knuth).

**톤**: 학습 카테고리. 정의 + 의사코드 + Python 예시 + 실전 코드 어디서 봤는지.

**예시 글감**:
- 워터마크 = 단조 시퀀스의 마지막 위치
- IRIS의 `paginationInfo` 처리에 무의식적으로 들어간 누적 합
- 다익스트라 — 의미와 한국 지하철 노선도 비유

**분량**: 1500~3500자.

**발행 풀**: `light/` 또는 `deep/`.

## 6. 운영 일지 (`ops-log`)

**무엇이 들어가나**: 만든 도구들이 그날그날 어떻게 굴러가고 있는지의 짧은 기록. 깨진 거 고친 거, 새 사이트 개편 대응, 일주일치 누적 다운로드량, 데이터 발견. **일기스러운 카테고리**.

**톤**: 가볍게, 짧게. 1인칭 평어체 그대로. 절대 경로 같은 식별 단서는 그대로 두지 말고 일반화.

**예시 글감**:
- 오늘 SMTECH가 사이트 개편으로 깨졌다. 30분만에 고침.
- 이번 주 누적 다운로드 12건. 그중 진짜 봐야 할 건 2건.
- 나라장터 API 일일 한도에 처음 도달한 날.

**분량**: 300~1000자.

**발행 풀**: 주로 `light/`. 일주일에 1~2편이 자연스러움.

## 7. 도구 사용기 (`tool-reviews`)

**무엇이 들어가나**: Claude Code 외에 작업에 사용한 도구들 짧은 후기. 에디터, CLI 도구, 라이브러리, 서비스, 윈도우 환경 도구 등. 한 글에 한 도구.

**톤**: 카메라 리뷰·문구 리뷰처럼 가벼움. 좋은 점·아쉬운 점·언제 쓰면 좋은가.

**예시 글감**:
- Streamlit으로 대시보드를 1시간만에 만든 후기
- `pyhwpx`로 한글 자동화 — 한컴오피스 의존의 현실
- Windows PowerShell + `chcp 65001` — 한국어 콘솔 인코딩 정착기

**분량**: 800~2000자.

**발행 풀**: `light/`.

## 8. 질문 노트 (`questions`)

**무엇이 들어가나**: 작업 중 막혀서 Claude(또는 검색·문서)에게 던진 **한 가지 질문**과 그 답에서 배운 한 줄. 가장 짧고 자주 올릴 수 있는 카테고리.

**톤**: 짧음. "질문 → 답 → 한 줄 배움" 3단 구조.

**예시 글감**:
- `requests.Session()` 첫 호출이 빈 응답을 준다 — 왜?
- ZIP 파일명이 일부만 깨진다 — `flag_bits & 0x800`의 의미
- SQLite의 UPSERT, `ON CONFLICT` 한 줄 사용법

**분량**: 200~700자. 일주일에 여러 편 가능.

**발행 풀**: `light/`.

## 9. 정보/코딩 교육 (`coding-education`)

**무엇이 들어가나**: 아동·청소년 코딩 교육의 단계별 주제. 다음 영역을 시간 순으로 또는 난이도 순으로 풀어 간다.

| 단계 | 주제 |
|---|---|
| 입문 | 코딩 기초 알고리즘 (순서·반복·조건), 블록 코딩(Scratch·엔트리) |
| 초급 | 텍스트 코딩 입문(Python), 어린이 바이브 코딩 — 자기가 좋아하는 주제로 사이트·게임·도구 만들기 |
| 중급 | AI 비전 인식(이미지 분류·객체 탐지·웹캠), 챗봇 만들기, 언어 모델 기초 |
| 하드웨어 | 아두이노 → 라즈베리파이 → (먼 미래) Nvidia Jetson Nano |

**톤**: 학습 친화. 아이가 보호자와 같이 따라 할 수 있는 단계 분해. 보호자·교육자도 부차 독자(PERSONA.md § 3).

**"어린이 바이브 코딩" 특별 노트**: 아이들이 좋아하는 주제(자기 캐릭터·반려동물·게임·이모지 등)를 Claude Code에게 "이런 거 만들고 싶어"라고 자연어로 말하고, 그 결과를 같이 다듬어 가는 과정. 코드를 가르치는 게 아니라 **자기 의도를 정확히 말하는 능력**과 **결과를 검증하는 능력**을 기르는 것이 목표. 이 시리즈가 본 블로그의 핵심 주제(§ 1)와 가장 일관됨.

**예시 글감**:
- 8살 짜리와 Scratch로 미로 게임 만들기 — 알고리즘 첫걸음
- 어린이 바이브 코딩 입문 — 자기 이름이 들어간 인사 사이트 만들기
- Raspberry Pi로 우리집 화분 물주기 알리미 (LED + 토양 센서)
- AI 비전으로 "내 강아지 사진 모아주는 도구" 만들기

**분량**: 1500~3500자. 단계별 캡처·코드·결과물.

**발행 풀**: 주로 `deep/`. 짧은 활동 후기는 `light/`.

## 10. AX (AI Transformation) (`ax`)

**무엇이 들어가나**: 제안서 작성·평가·공모를 준비하는 과정에서 학습한 주제들을 모은다. 정부 AX 사업의 공모 흐름, RFP 분석에서 자주 마주치는 평가 요소, 국방·공공·산업 영역의 AI 도입 사례 패턴, 도메인별 AI 적용 분석.

**톤**: 스터디 노트. 한 주제에 한 글. 신문 기사 정리가 아니라 **분석가의 관찰**에 가까운 톤. 출처(공고문·기사·정부 보고서·산업 보고서)는 본문 또는 각주에 명기.

**식별 단서 주의**: 본인이 작성 중인 특정 사업명·발주처·기업명은 일반화한다 (PERSONA.md § 2 화자 정체성 원칙). "어떤 부처의 AX 사업", "수요기관 A" 식으로 표현. 단, 공개된 공모·기사·연구 보고서를 인용할 때는 출처 그대로 표기 가능.

**예시 글감**:
- AX-Sprint 공모의 형식과 평가 배점 구조 분석
- RFP 평가표의 다섯 축 — 기술성·도메인 적합성·상용화·수행 역량·보안
- 국방 AI 도입의 전형 패턴 — "AI 참모", 의사결정 지원 시스템
- 디지털 트윈 + AI 조합의 실제 적용 사례 분석
- 공공 AI 사업에서 자주 등장하는 데이터 거버넌스 요건

**분량**: 1500~3500자.

**발행 풀**: 주로 `deep/`. 짧은 관찰 노트는 `light/`.

## 카테고리 외 — 태그(`tags`) 운용

카테고리는 "어디에 둘 글인가". 태그는 "이 글에 무엇이 들어 있나". 자유롭게 다는다.

자주 쓰는 태그 예: `claude-code`, `python`, `windows`, `crawling`, `rag`, `hwp`, `streamlit`, `sqlite`, `scratch`, `arduino`.
