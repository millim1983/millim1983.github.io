# 글쓰기 원칙 — Persona & Style Guide

이 블로그의 모든 게시물은 아래 원칙을 일관되게 따른다.
새 글을 시작할 때마다 이 문서를 먼저 확인할 것.

---

## 1. 핵심 메시지 (모든 글의 축)

> **"Claude Code는 엑셀이 함수에 한 일을 코드에 한다. 도구가 코드를 대신 짤 때, 사용자가 잘해야 하는 일은 — 무엇을 만들지 정의하는 능력과, 도구가 만든 결과를 검증하는 능력이다."**

이 명제가 모든 글의 결론으로 수렴한다. 코드 자체의 묘기를 자랑하는 글이 아니며, 코드 작성을 폄하하는 글도 아니다. 도구가 코드를 자동으로 작성해 주는 환경에서 **사용자의 일이 어디에 자리 잡는지**를 사례로 증명하는 글이다.

따라서 각 글의 구성에서 코드는 **증거**의 역할을 한다. 본문의 30~40 %를 차지하되, 나머지 60 % 이상은 다음에 할애한다.

- 어떤 문제를 풀어야 하는지 식별한 과정
- 여러 접근법 중 무엇을 왜 선택했는지의 판단
- AI가 자동으로 한 부분과 사람이 개입한 부분의 구분
- 그 사례에서 도출되는 일반적 시사점

## 2. 화자 (식별 단서를 남기지 않는다)

- 1인칭 단수는 **"나"**. (이전 안에서 사용했던 "필자"는 회피 — 너무 격식적이라 말투와 어긋남.)
- **직업·소속·도메인 단서를 글에 노출하지 않는다.** "PM이다", "○○업계 종사자이다", 회사명·도메인 이메일 등 일체 금지.
- 화자의 정체성은 단 하나로 충분하다: **"코딩이 본업이 아닌 실무자이지만, Claude Code라는 도구로 자기 업무에 필요한 자동화를 만들어 운영하는 사람."**
- "개발자 / 비개발자"의 이항 구도는 회피한다. 코드를 본업으로 다루는 사람들의 작업을 폄하하는 인상을 줄 수 있고, 시리즈의 정체성도 그 구도가 아니라 **도구 사용자의 자리**에 있다. 자기 호칭이 필요할 때는 "실무자" 또는 "코딩이 본업이 아닌 사람"으로 표현한다.
- 독자가 화자의 신원을 추정할 수 있는 어떤 단서도 남기지 않는다. 프로젝트 사례를 들 때도 회사·기관·고유 업무 맥락은 일반화하여 표현한다.

  | 회피 | 권장 |
  |---|---|
  | "제안서 업무에서 ~" | "특정 업무에서 ~" 또는 그 맥락 자체를 생략 |
  | "smartcoop.kr 이메일" | (언급하지 않음) |
  | "협동조합", "공공 분야" 등 분야 식별자 | 일반화 ("어떤 조직", "실무 현장") |

## 3. 독자

- **주된 독자**: 코딩이 본업이 아니지만 자기 업무에 쓸 도구를 직접 만들어 운영해 보고 싶은 실무자.
- **부차 독자**: 미래 세대(아동·청소년)에게 무엇을 가르쳐야 할지 고민하는 보호자·교육자.
- 전제하지 않는 것: 컴파일러·자료구조 지식, 특정 프레임워크 숙련도.
- 전제하는 것: 한국어 기술 문서 독해, 일반 디지털 도구 사용 능력.

## 4. 문체

- **종결어**: 평어체 (`-다 / -한다 / -이다`).
- **인칭**: "나" (1인칭 단수). "필자"는 안 쓴다.
- **AI 지칭**: "Claude Code" 또는 "Claude". 일반론에서는 "코딩 에이전트", "AI".
- **톤 기조**: 기술 블로그의 명료한 평어체. 격식보다는 **정확성**을 우선한다. 회의록·문서 자료에 인용해도 될 정도의 정확성을 유지하되, 문장이 딱딱하지 않게 적당한 호흡으로 읽히게 한다.

### 4-1. 어휘 원칙

| 회피 | 권장 |
|---|---|
| "때려보다" | "호출하다", "요청을 보내다" |
| "삽질" | "시행착오", "탐색" |
| 은어·축약어 ("쩐다", "갓-", "○년차") | 사용 안 함 |
| 추측형 ("~인 것 같다") | 단정형 ("~이다") — 근거가 약할 때만 "~으로 추정된다" |
| "막", "그냥", "쉽게" 같은 구어 부사 | 제거 |

전문 용어는 첫 등장 시 한 줄 정의를 괄호로 부기한다.
예: "AJAX 엔드포인트(브라우저가 동적으로 호출하는 서버 측 API)에 직접 요청을 보냈다."

### 4-2. 구조 (모든 글의 권장 골격)

1. **문제 식별** — 이 작업이 왜 필요했는가. 1~2 단락.
2. **선택지 비교** — 어떤 방법이 있었는가. 무엇을 왜 골랐는가. 사용자의 판단이 드러나는 핵심 구간.
3. **실행 과정** — 핵심 코드 인용. 도구가 자동으로 한 부분과 사용자가 개입한 부분을 구분 표시.
4. **결론 — 사용자의 일이 자리 잡는 곳** — 본 사례에서 도출되는, Claude Code 시대에 사용자가 갖춰야 할 능력은 무엇인가.

## 5. 도구/사용자 구분 표기 (모든 글에서 의무)

본문에서 작업을 묘사할 때 다음 표기를 일관 적용한다.

- **도구가 한 일**: "Claude Code가 ~을 자동으로 생성했다", "도구가 ~을 제안했다"
- **사용자가 한 일**: "나는 ~을 결정했다", "나는 ~의 문제를 식별하여 ~을 추가로 지시했다"

회색 영역(도구가 제안했고 내가 수용한 것)도 명시한다.
예: "Claude Code의 제안을 받아들였다. 다만 ~의 위험을 인지하고 있었기에 별도 검증을 거쳤다."

## 6. 코드 인용 규칙

- 코드 블록 위에 항상 **출처 파일 경로**를 표기. 단, 회사 또는 개인 식별이 될 수 있는 절대 경로는 상대 경로로 일반화한다.
  - 회피: `C:\Users\kbj\OneDrive\바탕 화면\biz-announcement-monitor\sites\iris\crawler.py`
  - 권장: `sites/iris/crawler.py:111-129`
- 핵심 줄에는 한국어 행 주석을 곁들여 비개발자도 흐름을 따라갈 수 있게 한다.
- 코드 인용 전에 **"이 코드를 보이는 목적이 무엇인가"**를 한 문장으로 먼저 밝힌다.

## 7. 매 글 말미에 포함하는 메타

> **AI 기여도**: 본 글이 분석하는 시스템 코드는 Claude Code가 약 _NN_ %를 작성하였다. 나는 요구사항 정의·설계 의사결정·검증을 담당하였다.
>
> **글 자체의 작성**: 초안 _N_ 회 / 직접 교정 _N_ 회.

수치는 정직하게 기재한다. 부풀리지 않는다.

## 8. 퇴고 체크리스트

- [ ] 평어체(-다 종결)로 일관되어 있는가
- [ ] "나" / "Claude Code" / "도구" 호칭이 일관되는가 ("필자" 잔재 없는가)
- [ ] 직업·소속·도메인 식별 단서가 노출되지 않는가
- [ ] 도구 수행 부분과 사용자 판단 부분이 명시 구분되었는가
- [ ] 코딩이 본업이 아닌 실무자가 따라갈 수 있는 어휘·예시인가
- [ ] **"사용자의 일이 어디에 자리 잡는가"가 결론에 분명히 들어갔는가** (핵심 메시지 검증)
- [ ] "개발자/비개발자" 이항 구도나 코드 폄하 표현이 남아있지 않은가
- [ ] 독자에게 시비 거는 듯한 단언·도전 표현이 남아있지 않은가
- [ ] 은어·구어·추측형 표현이 남아있지 않은가
- [ ] 절대 경로·이메일·고유 식별자가 본문에 남아있지 않은가
