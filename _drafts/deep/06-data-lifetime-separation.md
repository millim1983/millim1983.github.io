## 0. 폴더 하나가 두 종류의 시간을 담을 때

7편이 같은 데이터의 출처를 한 곳으로 줄이는 결정이었다면, 이번 회차는 같은 시스템 안의 데이터를 **수명과 관리자**에 따라 폴더를 다시 그리는 결정이다.

자동화가 운영 1년차에 한 번씩 마주치는 사고가 있다. 사용자가 디스크 청소를 하려고 다운로드 폴더를 비웠는데, 그 안에 시스템이 다시 안 쓰면 안 되는 state 파일까지 같이 들어 있어서, 다음 실행에 모든 중복 방지가 풀려 똑같은 파일을 다시 처음부터 받기 시작하는 경우다.

> **사용자는 "다운로드 폴더는 비워도 되는 것"이라고 안다. 시스템은 "이 폴더 안에 다신 못 만드는 게 있다"는 사실을 모른다.**

데이터의 수명을 폴더 경계로 드러내지 않으면 이런 사고가 반복된다.

## 1. 운영 요구를 한 문장으로 명문화

이 회차의 출발점은 운영 요구를 한 문장으로 적은 것이었다.

> "다운로드 폴더 안의 공고 폴더는 사용자가 임의로 비워도 다음 실행이 망가지지 않아야 한다."

이 문장이 폴더 설계의 상한선이 됐다. 어떤 데이터를 어디에 둘지 판단할 때마다 이 문장에 비추어 결정한다.

## 2. 두 종류의 데이터

자동화가 만들어 내는 데이터를 들여다보면 크게 두 종류로 갈린다.

| 종류 | 예시 | 수명 | 관리자 |
|---|---|---|---|
| **임시 데이터** | 다운로드한 PDF·HWP·중간 JSON·로그 | 짧음 (필요할 때만) | 사용자가 임의로 비울 수 있다 |
| **영구 데이터** | state(중복 방지)·이력·ledger·설정 | 시스템 수명과 동일 | 시스템만 관리. 사용자가 비우면 안 됨 |

이 두 종류가 같은 폴더 안에 섞여 있는 게 문제다. 분리해서 폴더 경계를 다르게 그어야 사용자가 임시 데이터만 안전하게 비울 수 있다.

## 3. 결과 구조 — `paths.py` 한 파일

도구가 만들어 낸 결과는 한 파일에 모인 6줄이다.

`core/paths.py` (요지)

```python
OUTPUT = Path(...)                                # 모든 산출물의 루트
DOWNLOADS = OUTPUT / "_downloads"                 # 첨부 폴더 — 비워도 안전
STATE_DIR = OUTPUT / "_state"                     # 이력·중복방지 — 절대 자동 삭제 X
LEDGER = STATE_DIR / "_ledger.xlsx"               # 통합 결과 표면
LOGS = OUTPUT / "_logs"                           # 로그 — 비워도 안전
TMP = OUTPUT / "_tmp"                             # 작업 중간 파일 — 비워도 안전
```

폴더 이름 앞에 언더스코어를 붙인 건 의도다. **사용자가 폴더를 탐색할 때 "시스템 폴더"임을 한눈에 알아보게** 한다. 일반 다운로드 폴더와 시각적으로 구분된다.

핵심 규칙은 두 줄이다.

- `_state/` 는 **절대 자동 삭제하지 않는다.** 디스크 청소 스크립트도 이 폴더는 건드리지 않는다.
- 그 외(`_downloads/`·`_logs/`·`_tmp/`)는 **사용자가 비워도 다음 실행에 영향이 없도록 모든 코드가 그 가정 위에서 동작**한다.

## 4. 자동 마이그레이션 — 이전 구조에서 안전하게 옮기기

이 분리를 처음부터 한 게 아니다. 운영 6개월차에 이전 구조의 위험을 인지하고 도입했다. 이전 구조는 다음과 같았다.

```
output/
├── _downloads/
│   ├── 공고-001/
│   ├── 공고-002/
│   └── _state.json    ← 다운로드 폴더 안에 state가 같이 있었다
```

이 구조에선 사용자가 `_downloads/` 를 통째로 비우면 `_state.json` 도 같이 날아간다. 다음 실행이 "처음 도는 척"이 되고, 중복 방지 정보가 사라져 같은 PDF를 다 다시 받기 시작한다.

Claude Code에게 다음 한 줄을 지시했다.

> "이전 위치(`_downloads/_state.json`)에서 새 위치(`_state/state.json`)로 자동 이동하는 마이그레이션 함수를 만들어. 시스템 시작 시 한 번 검사하고, 있으면 옮기고, 옮긴 뒤엔 이전 위치를 정리."

도구가 만들어 낸 결과는 `_migrate_legacy()` 함수 한 개. 시스템 부팅 시 한 번만 실행되고, 이전 구조의 사용자도 다음 실행 한 번에 새 구조로 자동 전환된다.

`core/paths.py:25-61` 의 마이그레이션 핵심 5줄:

```python
def _migrate_legacy():
    legacy = OUTPUT / "_downloads" / "_state.json"
    if legacy.exists():
        STATE_DIR.mkdir(parents=True, exist_ok=True)
        shutil.move(str(legacy), str(STATE_DIR / "state.json"))
        # 이전 위치 정리 (빈 폴더면 삭제)
```

마이그레이션은 한 번만 돌고 끝난다. 이후엔 새 구조가 정상 흐름.

## 5. 도구가 한 일과 사용자가 한 일

### 5-1. Claude Code가 자동으로 한 일

- `paths.py`의 폴더 정의 6줄 작성.
- `_migrate_legacy()` 마이그레이션 함수 구현. 부팅 시 1회 실행 보장 로직.
- 기존 코드에서 절대 경로로 박혀 있던 자리들을 `paths.py`의 상수 참조로 교체.
- 디스크 청소용 helper 스크립트(`_downloads/`·`_logs/`·`_tmp/` 비우기) 작성.

### 5-2. 사용자가 결정한 일

- **운영 요구의 한 문장 명문화** — "다운로드 폴더는 비워도 다음 실행이 망가지지 않는다." 이 한 문장이 폴더 설계의 상한선이 됐다.
- **수명에 따라 폴더를 나눈 결정** — 임시(다운로드·로그·tmp)와 영구(state)를 다른 폴더로. 한 폴더에 섞어 두면 사용자가 청소 시 사고를 낸다는 위험 식별.
- **사용자 시각적 구분의 강제** — `_state/`·`_downloads/` 같이 언더스코어로 시작하는 폴더명을 시스템 폴더 표식으로 채택. 폴더 이름이 운영 정책의 일부가 됐다.
- **이전 사용자의 안전한 전환** — 새 구조로 그냥 바꿔 버리면 이전 운영자들의 state가 사라진다는 점을 인지하고, 자동 마이그레이션을 같이 설계.
- **디스크 청소의 의미를 코드 주석에 박기** — "이 폴더는 비워도 안전" / "이 폴더는 절대 자동 삭제 X" 라는 한 줄을 각 폴더 정의 옆에 박는다. 6개월 뒤의 자기 자신을 위한 결정 기록.

코드 줄 수로 보면 사용자가 결정한 부분은 폴더 6줄 + 주석 6줄이다. **그러나 그 12줄이 운영 1년차에 일어났을 사고를 막는다.**

## 6. 한 줄 일반화 — "수명이 다르면 폴더가 다르다"

이 회차의 일반화는 다음과 같다.

> **데이터의 수명과 관리자가 다르면, 폴더 경계가 같으면 안 된다. 폴더 이름은 운영 정책의 일부다.**

같은 원리가 데이터 자동화 바깥에서도 적용된다. 회사 공유 드라이브에서 "이번 분기 자료"와 "10년 이력 자료"를 같은 폴더에 두면 정리 시점에 둘 다 같은 위험에 노출된다. 가계부 앱이 "이번 달 영수증 사진"과 "장기 결제 설정"을 같은 저장소에 두면 사용자가 사진 정리를 하다가 결제 설정을 날리는 일이 일어난다.

**수명에 따라 보관 위치를 다르게 그리는 일은 시스템 설계의 한 부분**이고, 그 결정은 사람의 자리에 있다. AI는 묻지 않으면 한 폴더에 다 넣는다.

## 7. 이 단계에서 사용자에게 새로 요구되는 능력

코드 한 줄을 잘 짜는 능력이 아니다. **자기 시스템의 데이터가 몇 종류의 시간을 사는지 식별하는 능력**, 그리고 **각 종류의 관리자가 누구인지를 명문화하는 능력**이다.

엑셀이 함수를 자동화한 뒤로 사람은 "어떤 함수를 어디에 쓸지" 정하는 일을 해 왔다. Claude Code가 코드를 자동화한 지금, **데이터가 어디서 살고 누가 청소할지** 정하는 일이 사람의 일이 됐다. 일관된 자리 이동이다.
