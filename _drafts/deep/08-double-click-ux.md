## 0. 도구의 사용성은 명령줄 너머에서 결정된다

코드를 본업으로 다루지 않는 사람이 자기 업무용 도구를 만들 때, 시스템이 잘 굴러가느냐 마느냐는 마지막 한 단계에서 갈린다. **터미널을 열고 옵션을 외우는 단계.** 처음에는 호기롭게 외워 보지만, 한 달이 지나면 "어 그 명령어 뭐였더라"가 시작된다. 두 달이 지나면 도구 자체를 안 쓰게 된다.

> **도구를 안 쓰기로 한 결정은 도구를 만들 때 이미 정해진다.**

이 회차의 주제는 비개발자가 만든 자동화가 **매일의 운영에서 살아남도록** 만드는 일이다. 핵심은 한 줄로 요약된다 — 사용자가 보는 결정 지점을 1개로 압축한다.

## 1. 시나리오 명문화 — 사용자가 보는 한 번의 클릭

가장 먼저 한 일은 운영 시나리오를 한 문장으로 적는 것이었다.

> "매일 아침 출근 후 `run.bat`을 더블클릭한다. 결과는 `_ledger.xlsx` 한 파일 또는 `localhost:8501` 한 URL에서 본다. 그 외의 어떤 결정도 일상 운영에서 다시 내리지 않는다."

이 한 문장이 시스템 설계의 상한선이다. 어떤 옵션이든 이 한 문장 안에 들어가지 않으면 빠진다. Claude Code가 새 기능을 제안할 때마다 이 문장에 비추어 채택 여부를 정했다.

## 2. 더블클릭의 실체 — `run.bat`

`run.bat`는 Windows 더블클릭 한 번으로 모든 사이트 크롤링을 끝까지 돌리고 통합 결과를 만들어 내는 단일 진입점이다.

`run.bat` (요지, 10줄 발췌)

```bat
@echo off
chcp 65001 > nul                              :: 한국어 콘솔 인코딩
cd /d "%~dp0"                                 :: 스크립트 위치로 이동
call .venv\Scripts\activate.bat               :: 가상환경 활성화
python run_all.py --sites iris,smtech,nipa,keit
python -m core.ledger_build                   :: 통합 ledger 생성
if errorlevel 1 (
    echo [실패] 로그를 확인하라 & pause
)
```

핵심은 두 가지다. **인코딩 선처리(`chcp 65001`)**, 그리고 **실패했을 때만 사용자에게 멈춤(`pause`)**. 평상시 성공 흐름에서는 사용자가 어떤 키도 누를 필요가 없다. 실패할 때만 사람이 등장한다.

Claude Code는 `run_all.py`의 `argparse` 인터페이스(`--sites`, `--full-list`, `--dry-run` 등)를 자동으로 만들어 줬다. 옵션을 받는 코드는 다 도구가 썼다. **그러나 일상 운영에서 어떤 옵션을 기본값으로 박을지의 결정은 사람이 했다.** 매일 돌리는 흐름에서 옵션을 묻지 않게 만드는 일, 그게 더블클릭의 본질이다.

## 3. 결과 확인 표면을 1개로 압축

자동화의 두 번째 함정은 결과의 산만함이다. 사이트별 어댑터가 각자 산출물(원본 PDF·HWP·중간 JSON·로그)을 자기 폴더에 떨군다. 자동화가 잘 돌아갈수록 사용자가 들여다봐야 할 파일이 늘어난다.

해법은 **사이트별 산출물은 그대로 두되, 사용자가 정식으로 확인하는 표면만 1개로 압축**하는 것이었다.

| 시스템 | 통합 표면 | 위치 |
|---|---|---|
| A — 정부 R&D 공고 모니터 | `_ledger.xlsx` 한 파일 | `output/_state/_ledger.xlsx` |
| B — 나라장터 입찰·낙찰 분석 | Streamlit 4탭 단일 URL | `http://localhost:8501` |

`_ledger.xlsx`의 컬럼 구조는 다음 한 줄로 압축된다.

| 컬럼 | 의미 |
|---|---|
| `site` | 출처 사이트 코드(iris/smtech/...) |
| `item_id` | 사이트별 고유 ID |
| `title` | 공고 제목 |
| `rcve_end_de` | 접수 마감일 |
| `status` | 신규 / 기존 / 마감 |
| `attach_path` | 다운로드된 첨부 경로(있을 때) |

매일 아침 사용자가 여는 파일은 이것 하나다. 사이트마다 따로 폴더를 열어 비교하지 않는다.

> **결과 확인 표면이 여러 개면 사용자는 그중 하나도 보지 않게 된다.**

Streamlit 쪽도 같은 원칙을 따랐다. 1탭(개요)·2탭(낙찰 패턴)·3탭(기관별)·4탭(키워드 검색). 4개 이상 늘리지 않았다. 더 보고 싶은 게 생기면 기존 탭 안에서 표나 차트를 갈아 끼운다.

## 4. 메타 자동화 — 사람의 수고 자체를 자동화 대상으로 보기

가장 흥미로운 진전은 **자동화 도구를 운영하는 사람의 수고까지 자동화 대상으로 본 것**이었다.

### 4-1. 반복 yes 클릭을 미리 허용하기

Claude Code는 셸 명령·파일 쓰기·외부 호출 같은 작업마다 사용자에게 권한을 묻는다. 처음에는 좋은 안전장치다. 그러나 매일 같은 명령을 yes로 통과시키는 게 하루의 일과가 되면, 이미 안전장치는 형해화돼 있다. yes를 빨리 누르는 게 일이 되니까.

해법은 `settings.local.json`에 **반복적으로 허용하는 것들을 미리 명시**하는 것이었다.

`.claude/settings.local.json` (요지)

```json
{
  "permissions": {
    "allow": [
      "Bash(python:*)",
      "Bash(.venv/Scripts/activate.bat:*)",
      "Bash(git status:*)",
      "Bash(git diff:*)",
      "Bash(git log:*)",
      "Read(./**)",
      "Edit(./**)"
    ]
  }
}
```

위험한 항목(예: `git push`, `rm -rf`, 외부 결제 API)은 일부러 빠져 있다. 사람의 결정이 필요한 자리에는 사람을 부르고, 매일 반복되는 안전 항목은 미리 통과시킨다. **권한을 풀어 주는 결정 자체가 사람의 일이고, 그 결정의 결과로 사람의 일상 노동이 줄어든다.**

### 4-2. 검증 OK 시 자동 진행, 실패 시에만 사람 부르기

크롤링 후 표본 검증 단계도 같은 원칙을 적용했다. 표본 5개를 골라 키 필드가 비어 있지 않은지, 인코딩이 깨지지 않았는지 자동 점검한다. 통과하면 다음 단계로 자동 진행. 한 개라도 실패하면 그 시점에 멈추고 사람에게 보고한다.

평상시 사용자가 보는 출력은 짧다.

```
[검증] OK (5/5)
[ledger] 새 항목 12개 추가
[done] 09:14
```

실패할 때만 사용자가 등장한다. 사람의 시간은 실패에만 쓴다.

### 4-3. "한 번의 클릭" 안에 사람의 결정이 압축되어 있다

겉으로 보면 더블클릭 한 번이지만, 그 안에는 다음과 같은 사람의 결정들이 미리 박혀 있다.

- 어느 사이트들을 매일 돌릴지(`--sites iris,smtech,nipa,keit`)
- 등록일 기준 early-stop을 켤지(`--full-list` 미사용 = 기본 켜짐)
- 표본 검증의 통과 기준(필드 5개, 비어 있으면 실패)
- 실패 시 어디서 멈추고 누구를 부를지(콘솔 `pause`, 로그 파일 경로)
- Claude Code가 자동 통과시켜도 되는 명령의 화이트리스트

각 결정을 일상 운영에서 다시 묻지 않도록 사전에 박아 두는 일, 이게 **메타 자동화**의 실체다. 비개발자가 만든 자동화가 두 달 뒤에도 살아 있는 이유는 거의 전부 여기에 있다.

## 5. 도구가 한 일과 사용자가 한 일

### 5-1. Claude Code가 자동으로 한 일

- `run.bat`의 인코딩·가상환경 활성화·에러 처리 셸 코드.
- `run_all.py`의 `argparse` 인터페이스, `--sites`·`--full-list`·`--dry-run` 옵션.
- `_ledger.xlsx` 빌더(`core/ledger_build.py`) — 사이트별 state를 모아 단일 표로.
- Streamlit `app_db.py`의 4탭 구조, 차트·필터·검색 컴포넌트.
- `settings.local.json` 권한 항목 후보 추천(어떤 명령을 자주 허용 요청하는지 분석).

### 5-2. 사용자가 결정한 일

- **시나리오 한 문장**: "매일 아침 더블클릭, 결과는 1개 표면에서 본다"라는 운영 시나리오를 명문화하고 그 한 문장으로 모든 기능 추가 요청을 거른 것.
- **결과 확인 표면을 1개로 압축**: 사이트별 산출물이 따로 있어도 사용자가 보는 정식 산출물은 통합 파일 하나로 한정한다는 결정.
- **권한 화이트리스트의 한도**: `settings.local.json`에 무엇을 풀고 무엇을 남길지의 선. `git push`나 외부 호출은 일부러 묻도록 남기는 판단.
- **사람을 부르는 시점의 정의**: 평상시에는 사용자가 등장하지 않고, 검증 실패·외부 장애·사이트 개편 같은 예외에만 사람을 부르는 보고 구조.
- **운영 시나리오를 검증하는 일과**: 한 달에 한 번쯤 일부러 옵션 없이 `run.bat`을 더블클릭해 보고, 새로 추가된 어떤 기능이 이 시나리오 한 문장을 깨지 않았는지 확인.

키보드 노동의 비중은 도구 쪽이 크다. **그러나 일상 운영에서 사람이 다시 결정하지 않도록 만드는 일의 비중은 사용자 쪽이 훨씬 크다.**

## 6. 이 단계에서 사용자에게 새로 요구되는 능력

이전 회차들의 결론이 "한 번의 호출을 성공시키는 능력"이었다면, 이 회차의 결론은 한 단계 위에 있다. **자기 자신의 일상 노동까지 자동화의 대상으로 보는 능력**이다.

Claude Code가 코드를 자동으로 써 주는 환경에서, 사용자의 시간은 "코드를 쓰는 시간"에서 "도구를 부리는 시간"으로 옮겨 갔다. 만약 도구를 부리는 시간이 그대로 매일 반복되는 yes 클릭과 옵션 입력으로 쌓이면, 자동화의 이득은 그만큼 갉아먹힌다.

> **자동화는 일상 운영에서 사람이 다시 결정하지 않도록 만드는 데까지 가야 끝난다.**

엑셀이 함수 계산을 자동화한 뒤로 사람의 일은 "어떤 함수를 어디에 쓸지" 정하는 자리로 옮겨 갔다. Claude Code가 코드 작성을 자동화한 지금, 사람의 일은 **"어떤 결정을 사전에 박아 두고, 어떤 결정만을 일상에서 다시 내릴지"** 정하는 자리로 옮겨 간다. 한 단계 더 올라간 같은 종류의 자리 이동이다.

## 7. 다음 회차 예고

다음 회차는 같은 결정의 또 다른 모습을 다룬다. 같은 정보가 여러 사이트에 동시에 올라올 때, **어느 출처를 진실원(SSOT)으로 삼을지** 결정하는 일이다. AI는 "어떻게 수집할 것인가"는 다 풀지만 "어디서 수집할 것인가"는 묻지 않는다.
