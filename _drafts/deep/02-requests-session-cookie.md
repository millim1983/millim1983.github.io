## 0. 시작점 — 브라우저는 잘 되는데 코드는 빈 응답을 준다

1편에서 IRIS 사이트의 AJAX 엔드포인트를 직접 호출하기로 결정한 다음, Claude Code는 `requests.post(...)`로 변환한 호출 코드를 단숨에 만들어 주었다. 첫 실행 결과는 다음과 같았다.

```python
{"resultList": [], "paginationInfo": {"totalRecordCount": 0}}
```

같은 시각 브라우저로 같은 화면에 들어가면 공고 목록이 정상적으로 떴다. 같은 URL·같은 HTTP 메서드·같은 파라미터인데 한쪽은 데이터를 받고 한쪽은 빈 배열을 받는다.

> **같은 HTTP 요청처럼 보여도 서버가 보기엔 다른 요청일 수 있다.**

이 회차의 주제는 그 차이를 어떻게 한 줄씩 비교해서 좁혀 가느냐다.

## 1. 두 가지 가설

빈 응답을 받았을 때 떠올린 가설은 두 가지였다.

| 가설 | 내용 | 시험 방법 |
|---|---|---|
| ① | 요청 파라미터가 부족하다. 화면이 보내는 숨은 파라미터를 빠뜨렸다. | 브라우저 Network 탭에서 같은 호출의 요청 본문을 그대로 복사해 보냄. |
| ② | 세션 쿠키가 없다. 첫 진입 시 발급되는 `JSESSIONID` 같은 식별자 없이 AJAX만 직접 친 게 문제다. | 화면 진입 GET 요청을 한 번 보낸 뒤 같은 세션으로 AJAX를 다시 호출. |

Claude Code가 처음에 생성한 코드는 ①의 위험을 줄이는 방향이었다. `requests.post(...)`에 캡처한 파라미터를 그대로 채워 넣었으니까. 그런데도 빈 배열이 왔다는 건 ①의 가설이 깨졌다는 뜻이다. ②를 먼저 시험할 차례였다.

## 2. 진단 절차 — "두 요청의 차이를 한 줄씩 비교한다"

여기서 사용자(나)의 일은 단순한 절차였다. **브라우저에서 성공하는 요청과 코드에서 실패하는 요청, 두 개의 헤더·쿠키를 줄 단위로 비교한다.** 도구를 부르기 전에 사람이 먼저 해야 할 일이다.

브라우저 쪽 요청에는 다음이 들어 있었다.

```
Cookie: JSESSIONID=xxxxxxxxxxxxxxxxxxxxxxxx; ...
Referer: https://www.iris.go.kr/contents/retrieveBsnsAncmBtinSituList.do
```

코드 쪽 요청에는 둘 다 없었다. **브라우저는 그 화면에 들어간 시점에 GET 요청을 한 번 보냈고, 그때 서버가 발급한 `JSESSIONID`가 이후 모든 AJAX 호출에 자동으로 따라붙고 있었다.** 코드는 첫 GET을 생략한 채 AJAX부터 쳤으니 서버 입장에선 "어디서 온 누군지 모를 호출"이었다.

이 시점에 Claude Code에게 한 줄 지시를 보냈다.

> "AJAX 호출 전에 화면 GET을 한 번 보내고, 같은 `requests.Session()`으로 이어서 호출하도록 고쳐줘. `Referer` 헤더도 함께 붙여."

## 3. 결과 코드 — `new_session()`

도구가 만들어 낸 결과는 5줄 안에 끝난다. 흐름이 보이도록 한국어 주석을 곁들였다.

`sites/iris/crawler.py:72-77`

```python
def new_session():
    s = requests.Session()                    # 쿠키 자동 보관
    s.headers.update({
        "User-Agent": "...",                  # 브라우저인 척
        "Accept-Language": "ko,en;q=0.8",
    })
    s.get(LIST_VIEW_URL, timeout=20)          # 화면 진입 — JSESSIONID 확보
    return s
```

핵심은 마지막 줄이다. `requests.Session()`은 호출 사이의 쿠키를 알아서 보관해 주지만, **첫 GET 자체를 생략하면 보관할 쿠키가 없다.** 한 줄을 추가한 것만으로 빈 배열이 정상 응답으로 바뀌었다.

AJAX 호출 측에서도 `Referer` 헤더를 같이 명시하도록 보강했다.

`core/http.py:7-12` (요지)

```python
COMMON_HEADERS = {
    "User-Agent": "...",
    "Accept": "application/json, text/javascript, */*; q=0.01",
    "Accept-Language": "ko,en;q=0.8",
    "X-Requested-With": "XMLHttpRequest",     # AJAX임을 명시
}
```

이렇게 정리해 두면 IRIS만이 아니라 SMTECH·NIPA·KEIT 같은 다른 AJAX 사이트에서도 같은 헤더 묶음을 재사용할 수 있다. 사이트별 어댑터는 `Referer`만 자기 화면 URL로 갈아 끼우면 된다.

## 4. 도구가 한 일과 사용자가 한 일

### 4-1. Claude Code가 자동으로 한 일

- `requests.Session()` 객체 생성과 보관 패턴.
- 브라우저급 `User-Agent`·`Accept-Language` 헤더 세팅.
- `Referer` 헤더 부여, `X-Requested-With: XMLHttpRequest` 추가.
- 공통 헤더를 별도 모듈(`core/http.py`)로 분리해 어댑터들이 공유하는 구조.

한 줄로 줄이면, **"고치라고 지시하면 정확히 그 코드를 쓴다."**

### 4-2. 사용자가 결정한 일

- **"왜?"를 끝까지 추적하기로 한 결정**: 빈 배열을 받았을 때 그냥 파라미터를 더 던져 보는 식의 시도를 멈추고, 두 요청의 차이부터 비교한다는 절차를 선택.
- **가설의 우선순위**: ① 파라미터 누락보다 ② 세션 쿠키 부재를 먼저 시험한다는 판단. 근거는 "Claude가 캡처한 파라미터를 그대로 옮겼는데도 빈 응답이라면, 다른 변수에서 차이가 났을 가능성이 높다"는 추정.
- **재현 절차 정리**: 브라우저 Network 탭 → 두 요청의 헤더·쿠키 줄 단위 비교 → 차이 한 줄 식별. 이 절차를 다음 사이트(SMTECH·NIPA·KEIT)에서도 그대로 재사용 가능하도록 자기 안에 정착시키는 일.
- **`Referer`까지 같이 보강**: 세션 쿠키만 추가해도 그 사이트는 동작했지만, 다른 사이트에서는 `Referer`가 결정적인 차이일 수도 있다는 점을 일반화해서 공통 헤더에 미리 넣어 둠.

키보드 노동량으로 보면 ①번 절차(차이 비교) 한 번에 30초도 안 걸린다. 그러나 그 절차를 시작할지 말지 결정하는 게 사람의 일이다.

## 5. 한 줄 일반화 — "디버깅의 출발점은 비교다"

이 사례에서 일반화할 수 있는 한 줄은 다음과 같다.

> **같은 HTTP인데 결과가 다를 때, 사람이 가장 먼저 하는 일은 "두 요청의 차이를 한 줄씩 비교하는 것"이다.**

AI는 그 비교의 결과로 도출된 한 줄짜리 코드 변경은 정확히 쓴다. 그러나 "지금이 비교를 시작할 시점이다"라는 판단은 묻지 않으면 하지 않는다. 새 파라미터를 던져 보거나, 더 그럴듯한 에러 처리 코드를 덧붙이거나, 다른 라이브러리를 권하는 식으로 우회한다. 비용은 크고 원인은 그대로 남는다.

이 회차에서 사용자가 갖춰야 할 능력은 **"두 요청의 차이를 한 줄씩 비교하는 절차"를 자기 안에 가지고 있는 것**이다. 절차 자체는 어렵지 않다. 가지고 있느냐 없느냐가 결과를 가른다.

## 6. 다음 회차 예고

3편은 같은 AJAX 호출이 **사이트별로 어떻게 다르게 생겼는지** 표 한 개로 정리한다. IRIS·SMTECH·NIPA·KEIT 네 사이트의 엔드포인트·파라미터·응답 키를 줄 세워 비교하면서, "비공식 인터페이스를 어디까지 써도 되는가"라는 한도의 문제까지 같이 다룬다.
