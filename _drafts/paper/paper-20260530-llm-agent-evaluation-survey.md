> **TL;DR** — Hebrew University · IBM Research · Yale 공동 서베이. LLM 에이전트 평가를 **5개 축**으로 정리하고, 대표 벤치마크 7~10개의 분류표를 제시한다. 가장 중요한 메시지는 "**LLM 능력과 에이전트 하네스 능력을 분리해서 평가하라**"는 한 줄이다. 트랙 B의 [멀티 에이전트가 default가 됐다] 글과 짝을 이뤄 읽으면 좋다.

## 1. 모델 한눈에 — 이 논문은 모델이 아니라 평가 체계를 만든다

이건 새 모델 논문이 아니라 **평가 방법론의 지도**다. 저자들은 LLM 에이전트 평가를 다섯 축으로 분류한다.

| 축 | 무엇을 본다 |
|---|---|
| ① 핵심 LLM 능력 | 계획·도구 사용·자기 반성·메모리 |
| ② 애플리케이션 특화 | 웹·소프트웨어 공학·과학·대화 |
| ③ 일반주의 에이전트 | 다목적 작업 |
| ④ 벤치마크 차원 | 데이터·환경·인터페이스·메트릭·안전성 |
| ⑤ 평가 프레임워크 | 개발자가 자기 에이전트를 평가하는 도구 |

이 다섯 축이 모두 따로 발전해 왔다는 점이 이 서베이의 출발점이다.

## 2. 데이터 준비 — 대표 벤치마크 7개

이 논문이 정리하는 주요 벤치마크 7개를 한 줄씩 보면 동네 풍경이 보인다.

| 벤치마크 | 도메인 | 평가하는 능력 |
|---|---|---|
| **SWE-bench Verified** | 소프트웨어 공학 | 실제 GitHub 이슈 해결 |
| **WebArena** | 웹 | 동적 환경에서의 행동 |
| **Mind2Web** | 웹 | 오프라인 시나리오 처리 |
| **GAIA** | 일반 AI 어시스턴트 | 다중 모드·복합 추론 |
| **τ-Bench** | 대화 + 도구 | 대화 흐름 안에서의 도구 사용 |
| **AppWorld** | 일반주의 에이전트 | 애플리케이션 환경 적응 |
| **SciCode** | 과학 | 코드 생성을 통한 과학 문제 풀이 |

벤치마크 간 **공통 척도가 없다**는 사실이 이 표 자체에서 드러난다. SWE-bench의 점수와 GAIA의 점수는 비교할 수 없다.

## 3. 학습 절차 — 평가 서베이라 N/A

이 논문은 학습을 다루지 않는다. 다만 **"평가가 학습을 견인한다"** 는 일반론은 강하게 깔려 있다. 벤치마크가 부실하면 모델 개발 방향이 잘못 잡힌다. 그래서 평가 체계 자체를 다지는 일이 모델 개선만큼 중요하다는 메타 주장이다.

## 4. 파인튜닝 — 다루지 않음

이 서베이는 파인튜닝 기법을 분류하지 않는다. 그러나 평가 결과를 봤을 때 **"파인튜닝된 모델 vs 범용 모델"의 비교가 부족하다**는 점을 미래 과제로 언급한다. 도메인 특화 에이전트의 가치를 정량화하려면 비교 가능한 벤치마크 설계가 먼저다.

## 5. 성능 평가 — 이 논문의 본체

본 서베이의 핵심이다. 저자들은 **벤치마크 자체의 다섯 차원**을 다음과 같이 정리한다.

### 5-1. 벤치마크의 다섯 차원

| 차원 | 무엇을 본다 |
|---|---|
| **데이터 큐레이션** | 시나리오를 누가 어떻게 만들었는가 |
| **환경** | 정적인가 동적인가, 대화형인가 |
| **상호작용 인터페이스** | API·UI·자연어·코드 |
| **메트릭** | 성공률·정확도·효율·안전 |
| **안전성** | 부작용·악용·규정 준수 |

지금까지 대다수 벤치마크는 위 다섯 중 1~2개에만 집중했다.

### 5-2. 5개의 미해결 문제

저자들이 콕 집은 미래 과제 5가지. 산업 도입 평가표를 짤 때 그대로 차용 가능하다.

1. **세분화된 평가 부족** — 현재 벤치마크는 종합 성공률에 의존, 중간 결정 과정 진단 불가
2. **비용 효율성 미측정** — 토큰·시간·리소스 소비가 평가에 거의 안 들어감
3. **안전성·규정 준수** — 대부분 기능 중심, 안전성 테스트 부족
4. **확장성 자동화** — 정적 인간 주석 데이터의 확장 한계
5. **LLM과 하네스의 혼동** — 백본 모델 능력과 에이전트 설계 능력을 분리해서 측정해야

특히 5번이 산업 도입자에게 가장 직접적인 메시지다. **"Claude를 썼는데 잘 됐다"** 가 모델 덕인지 우리가 만든 에이전트 하네스 덕인지 분리하지 못하면, 다음 모델로 갈아탔을 때 무엇이 깨지는지 모른다.

## 6. 우리는 이걸 어떻게 활용하면 좋겠는가 — 실무 함의

이 자리에 본 글의 절반을 쓴다.

### 6-1. RFP·평가표 작성의 직접 차용 가능한 5축

AX 사업이나 사내 AI 도입 평가표를 짤 때, **벤치마크의 다섯 차원**을 평가 항목으로 그대로 번역할 수 있다.

- 데이터 큐레이션 → "우리 사례 시나리오는 누가 어떻게 큐레이션할 것인가"
- 환경 → "정적 테스트 / 운영 환경 시뮬레이션 / 실제 사용자 베타 — 어느 단계까지 평가하는가"
- 상호작용 인터페이스 → "최종 사용자가 마주하는 인터페이스의 평가는 따로 있는가"
- 메트릭 → "비용·시간·정확도·일관성 중 어느 가중치인가"
- 안전성 → "오작동의 결과가 어디까지 영향을 미치는가 (금전·법적·물리적)"

다섯 항목을 평가표에 박는 일은 발주자의 의무가 됐다. 이 서베이가 그 정당성의 근거가 된다.

### 6-2. LLM 점수와 시스템 점수의 분리 측정

"Claude Code 도입 후 생산성 30 % 증가" 같은 보고가 흔해졌다. 이 서베이가 던지는 질문은 다음이다 — **그 30 %가 Claude 모델의 점수인가, 우리가 만든 명령·프롬프트·도구 호출 구조의 점수인가.**

분리해서 측정하지 않으면, Claude 4.7 → 4.8 업그레이드 또는 다른 모델로 전환했을 때 결과가 어떻게 바뀔지 예측 불가다. 모델 의존성을 정량화하는 일이 시스템 운영자의 일이 됐다.

### 6-3. 비용 효율성을 평가 1순위로 올리는 정당성

5개 미해결 문제 중 2번(비용 효율성 미측정)은 본 블로그가 일관되게 다루는 운영 시점의 관점과 정확히 맞물린다. 매일 cron이 도는 자동화의 경제성, AX 사업의 ROI 평가, 사내 LLM 도입의 토큰 비용 — 모두 학술 벤치마크가 다루지 않는 자리다.

> **벤치마크가 측정하지 않는 차원을 사용자가 자기 환경에서 측정한다.**

이 논문이 제공하는 가장 큰 정당성이다. "비용은 부차적"이라는 주장에 반박할 학술적 근거가 생겼다.

## 7. 한 줄 정리

> **에이전트 평가는 5축이다. 한 축짜리 점수에 도입 결정을 걸지 말 것.**

산업 도입자가 학술 벤치마크 자체를 액면 그대로 받아들이지 않고 **자기 환경의 평가표를 짜는 일**의 정당성을 학술적으로 확보한 서베이다.
