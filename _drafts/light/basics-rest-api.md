## 0. 한 줄 정의

**REST API(REpresentational State Transfer API)** — 서버에 있는 자원을 URL로 가리키고, HTTP 메서드(GET·POST·PUT·DELETE 등)로 그 자원을 어떻게 다룰지 약속한 인터페이스.

이름이 거창해 보이지만 핵심은 단순하다. **자원에 주소를 붙이고, 그 주소에 메서드를 보내서 일을 시킨다.** 그게 전부다.

## 1. REST 이전을 잠깐 본다

REST가 자리 잡기 전에는 하나의 URL이 여러 동작을 다 받았다. `?action=create`, `?action=update`, `?action=delete` 같은 파라미터로 동작을 구분했다.

```
POST /api/handler.php?action=update&id=42
```

같은 URL이 무엇을 하는지 본문이나 파라미터를 까봐야 알 수 있었다. 클라이언트도 서버도 약속이 매번 달랐다.

REST는 이걸 뒤집었다. **자원이 먼저, 동작은 메서드로.**

```
PUT /api/users/42
```

URL을 보면 자원이 무엇인지 한눈에 들어온다. 메서드를 보면 무엇을 하려는지가 드러난다.

## 2. 자원·URL·메서드의 짝

REST에서 URL은 자원의 주소다. 보통 복수형 명사로 컬렉션을 가리키고, 그 뒤에 식별자를 붙여 개별 자원을 가리킨다.

```
/users           ← 사용자 컬렉션
/users/42        ← 42번 사용자
/users/42/posts  ← 42번 사용자의 글 컬렉션
```

같은 URL에 어떤 메서드를 보내느냐로 동작이 결정된다.

| 메서드 | 컬렉션 (`/users`) | 개별 자원 (`/users/42`) |
|---|---|---|
| **GET** | 사용자 목록을 읽는다 | 42번 사용자 정보를 읽는다 |
| **POST** | 새 사용자를 만든다 | (보통 사용 안 함) |
| **PUT** | (보통 사용 안 함) | 42번 사용자 정보를 통째로 바꾼다 |
| **PATCH** | (보통 사용 안 함) | 42번 사용자 정보의 일부만 바꾼다 |
| **DELETE** | (보통 사용 안 함) | 42번 사용자를 삭제한다 |

이 짝이 일관되면, 다른 사이트의 API 문서라도 URL 모양만 봐도 어떤 호출을 어떻게 해야 하는지가 추측 가능하다. 그게 REST의 실용적 가치다.

## 3. 상태 코드로 결과를 알린다

서버가 응답할 때 본문만 돌려주는 게 아니라, **HTTP 상태 코드**로 결과 분류를 함께 알려준다. (상태 코드 한 편이 따로 있으니 여기서는 짧게.)

- `200 OK` — 잘됐다
- `201 Created` — 새 자원이 만들어졌다 (POST 응답에 자주)
- `204 No Content` — 잘됐는데 돌려줄 본문이 없다 (DELETE 응답에 자주)
- `400 Bad Request` — 요청이 잘못됐다
- `401 Unauthorized` — 인증이 안 됐다
- `404 Not Found` — 그 자원이 없다
- `409 Conflict` — 중복·충돌 등 상태 모순
- `500 Internal Server Error` — 서버가 터졌다

REST API를 호출했는데 응답이 이상하면, 본문보다 먼저 상태 코드를 본다. 본문은 거짓말을 할 수 있지만 상태 코드는 서버 프레임워크가 자동으로 정확히 박는 경우가 많다.

## 4. 실제 호출이 어떻게 생겼나

브라우저나 도구에서 본 REST 호출은 대체로 이렇게 생겼다.

```
GET /api/announcements/142 HTTP/1.1
Host: example.org
Accept: application/json
Authorization: Bearer eyJhbGciOi...
```

응답:

```
HTTP/1.1 200 OK
Content-Type: application/json

{
  "id": 142,
  "title": "공고 제목",
  "deadline": "2026-06-30"
}
```

URL은 자원을, 메서드는 동작을, 본문은 데이터를, 상태 코드는 결과를. 각자 자기 자리에서 자기 일만 한다.

## 5. 개발자 도구에서 REST 호출 읽기

`basics-ajax` 편에서 다룬 Network 탭 관찰법이 그대로 적용된다. REST 관점으로 볼 때는 주목할 칸이 따로 있다.

1. F12 → Network → Fetch/XHR
2. 요청 한 줄에서 **메서드**(GET/POST/PUT/...), **URL의 자원 부분**(`/api/users/42` 같은), **상태 코드**(200, 404 등)
3. 클릭하면 헤더·요청 본문·응답 본문을 다 볼 수 있다

이 셋만 봐도 그 사이트가 REST 규칙을 얼마나 잘 따르는지 보인다. 자원 이름이 명사인지, 메서드 의미가 일치하는지, 상태 코드가 적절한지.

## 6. REST가 답이 아닐 때

REST는 자원이 명확한 작업에 잘 맞는다. 다음 경우엔 다른 스타일이 더 낫다.

- **여러 자원을 한 번에 정밀하게 골라 받아야 할 때** — REST는 자원 단위가 기본이라 한 화면에 여러 자원이 필요하면 호출이 여러 번이 된다. GraphQL이 한 호출로 묶어준다.
- **실시간 양방향 통신이 필요할 때** — 채팅·알림처럼 서버가 먼저 말을 거는 경우엔 WebSocket이나 SSE 쪽이 맞는다.
- **명사로 안 떨어지는 동작** — "결제를 환불한다", "메일을 보낸다" 같은 동작은 자원으로 비틀어 표현하는 데 한계가 있다. 이 경우엔 RPC 스타일이 더 자연스럽다.

## 7. 한 줄 마무리

> **REST API = 자원에 URL을 붙이고, 거기에 HTTP 메서드를 보내서 동작을 시키는 약속.** 결과는 상태 코드로 알린다.

다른 사이트의 API 문서를 처음 봤을 때 URL의 자원 부분, 메서드, 상태 코드 이 셋만 먼저 짚어 읽으면 전체 구조가 거의 다 잡힌다. 외울 게 많아 보여도 약속의 모양은 늘 같다.
