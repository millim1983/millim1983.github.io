> **TL;DR** — 2026년 3월 15일 arXiv에 올라온 논문. arXiv 696,758쌍의 고/저인용 논문을 reward 신호로 써서 Qwen3 계열 모델을 학습시켰더니, GPT-5.2·Gemini 3 Pro 같은 상용 모델보다 "연구 아이디어의 잠재력 판단"을 더 잘하게 됐다는 결과를 내놨다. 새 사실보다 새 학습 신호 — **RLHF가 아닌 RLCF(Reinforcement Learning from Community Feedback)** — 가 이 논문의 진짜 기여다.

## 1. 모델 한눈에 — 판단 모델과 제안 모델, 두 마리

논문은 두 모델을 차례로 만든다.

| 모델 | 역할 | 베이스 |
|---|---|---|
| **Scientific Judge (SciJudge)** | 논문 두 편을 받아서 어느 쪽이 더 임팩트 있을지 판단 | Qwen2.5-Instruct (1.5B–32B), Qwen3 (4B, 30B), Llama-3.1-8B-Instruct |
| **Scientific Thinker** | 시드 논문(제목+초록)을 보고 후속 연구 아이디어 8개를 제안 | Qwen3-30B-A3B-Thinking, Qwen3-4B-Thinking |

SciJudge는 단순 분류기가 아니라 **생성형 reward model**이다. 추론 trace를 먼저 풀어 쓰고 그 끝에 선호를 예측한다. Scientific Thinker는 그 SciJudge를 reward 함수로 쓰는 정책 모델이다.

학습 인프라는 MS-SWIFT + DeepSpeed ZeRO-2/3, 모델 크기에 따라 H200 32~128장.

## 2. 데이터 — 인용수를 reward로 쓸 수 있게 가공한 방법

논문의 알맹이는 **SciJudgeBench**라는 696,758개 페어 데이터다. 그냥 인용수 비교가 아니라 시간·분야 편향을 제거한 비교다.

- **출처**: 인용 메타데이터 붙은 arXiv 논문 약 2.1M편, 페어 구성에 들어간 고유 논문 1.4M편
- **분야 매칭**: arXiv 서브카테고리 단위(`cs.CV`, `math.AP`, `quant-ph` 등)로 같은 영역끼리만 짝
- **시간 매칭**: 같은 발표 시기 윈도우 안에서만 짝 (오래된 논문이 인용수 누적상 유리한 효과 제거)
- **인용 임계값**: 절대 차이 ≥ 8회, 상대 차이 ≥ 30%. 테스트셋은 더 엄격하게 절대 32 / 상대 50%

분야 분포는 컴퓨터과학 161,461쌍, 수학 102,592쌍, 물리 394,336쌍, 기타 38,369쌍. 학습 시점 기준 2024년까지 발표분.

이 데이터 구성이 이 논문의 진짜 비용이고 진짜 기여다. "인용수 = 임팩트"라는 단순 가정이 시간·분야 편향에 의해 망가지는 걸 매칭 설계로 막은 부분.

## 3. 학습 절차 — GRPO에 이진 보상을 붙였다

SciJudge 학습은 RLHF 계열의 표준 도구를 가져왔다.

- **알고리즘**: Group Relative Policy Optimization (GRPO)
- **보상**: 이진(binary) — 정답 선호와 일치하면 1, 아니면 0
- **이점 정규화**: 그룹 내 정규화 `(r_i - mean(r)) / std(r)`
- **목적함수**: 클리핑된 surrogate + 참조 정책 KL 페널티 (β = 0.03)
- **학습량**: 696,758쌍 1 epoch, 배치 128

특이점은 PPO·DPO가 아니라 GRPO를 골랐다는 것과, 보상 신호가 사람 라벨이 아니라 **커뮤니티가 사후적으로 매긴 인용수**라는 것. 저자들은 이걸 RLHF와 구분해서 **RLCF (Reinforcement Learning from Community Feedback)** 로 부른다.

## 4. 파인튜닝 — SciJudge가 다시 Reward가 된다

Scientific Thinker는 SciJudge를 보상 모델로 써서 학습된다.

- **베이스 정책**: Qwen3-30B-A3B-Thinking, Qwen3-4B-Thinking (2507 버전)
- **보상 모델**: SciJudge-Qwen3-4B (생성형 reward)
- **알고리즘**: Comparison-Based GRPO + 라운드로빈 토너먼트
- **보상 계산**: 한 프롬프트당 8개 응답을 뽑고, 그들 사이 페어 비교 승률을 보상으로
- **태스크**: 시드 논문(제목+초록)을 주면 후속 연구 아이디어 1개를 제안
- **학습 데이터**: 2025년 1~7월의 고인용 논문 4,000편. in-domain·out-of-domain 테스트셋 각 200편
- **하이퍼파라미터**: learning rate 5e-7, β = 0.001, 8 generations/prompt

여기서 인상 깊은 건 학습 데이터가 4,000편이라는 점이다. SciJudge 696K에 비해 두 자릿수 작다. **판단 모델은 양으로, 제안 모델은 양보다 신호의 질로 — 그게 이 파이프라인의 형태다.**

## 5. 성능 평가 — 어떻게 일반화를 증명했나

평가 셋이 다섯 가지다. 일반화를 다섯 축으로 갈라 본다.

| 테스트셋 | 페어 수 | 어떤 일반화를 시험하나 |
|---|---|---|
| Main (In-Domain) | 728 | 같은 분포 |
| Temporal OOD | 514 | 2025년 발표분 (학습 시점 이후) |
| Metric OOD (ICLR) | 611 | 인용수 대신 ICLR 피어 리뷰 점수 |
| Field OOD | (분야별) | CS로 학습한 모델을 Math·Physics·기타에 |
| Biology OOD | 160 | bioRxiv — 학습에 없던 분야 |

핵심 결과: **SciJudge-Qwen3-30B가 평균 80.6%** 페어 정확도(위치 교차 일관성 포함)로 GPT-5.2·Gemini 3 Pro를 넘었다. Scientific Thinker는 in-domain 76~83% 승률.

Ablation: 같은 정책에 untrained reward(Qwen3-4B-Instruct)를 붙이면 in-domain 73.0% / OOD 70.5%. SciJudge로 바꾸면 81.5% / 83.0%. 보상 모델의 질이 정책 모델의 일반화에 그대로 옮겨간다는 신호다.

## 6. 실무 활용 함의 — 이 논문이 비-LLM 실무자에게 주는 것

논문 리뷰의 절반 이상을 여기에 쓴다. Claude Code로 자동화를 만드는 사용자의 자리에서 보면, 이 논문이 던지는 메시지는 세 갈래다.

### (가) "좋은 결과물"의 신호를 무엇으로 둘 것인가

가장 큰 함의. 이 논문은 "인용수"라는 사후 신호를 reward로 쓸 수 있게 가공하는 방법을 보여준다. 그대로 옮기면 다음 자리에 적용 가능하다.

- 사내 보고서·기술 문서의 품질 reward → "리뷰어 점수", "재인용 횟수", "검색 클릭률" 같은 사후 신호
- 코드 PR 품질 reward → "리뷰어 승인까지의 round 수", "PR 머지 후 30일 내 버그 fix 횟수"
- 입찰 제안서 품질 reward → "수주 여부", "심사위원 점수 분포"

핵심은 사후 신호를 그대로 쓰는 게 아니라, **시간·분야·평가자 편향을 매칭으로 깎아내고 나서** reward로 쓰는 과정이다. SciJudgeBench의 진짜 가치는 데이터 규모가 아니라 그 매칭 설계에 있다.

### (나) 판단 모델과 제안 모델을 분리하는 구조

"좋은 걸 만드는 모델"을 직접 학습하려 들지 말고, "좋은 걸 알아보는 모델"을 먼저 만든 다음 그것을 보상으로 쓰라는 패턴. 사용자가 사내에서 LLM 응용을 만들 때 적용 순서가 명확해진다.

1. 먼저 **판단 모델**을 데이터 많이 부어 학습 (사후 신호 + 매칭)
2. 그 판단 모델을 **reward 함수**로 고정
3. 그 위에서 **제안 모델**을 작은 데이터로 RL 학습

이 구조는 코드 자동화에도 응용 가능하다. "어떤 코드가 PR 통과율이 높은가"를 학습한 판단 모델이 있으면, 그것을 reward로 코드 제안 모델을 fine-tuning할 수 있다.

### (다) 한계 — 인용수를 신뢰할 수 있을 때만 작동한다

이 파이프라인은 사후 신호가 충분히 누적된 도메인에서만 의미가 있다. 사내 문서·내부 자동화 같이 사후 신호가 적거나 노이즈가 심한 자리에선 RLCF 그대로는 안 굴러간다. 대안으로 논문 저자들은 ICLR 피어리뷰 점수 같은 다른 사후 신호로의 일반화를 보여줬는데, 사용자 자신의 도메인에서 어떤 신호가 "재인용수"의 자리를 대신할 수 있는지 먼저 답해야 한다.

이 자리가 사용자의 일이다. 도구가 reward 함수를 직접 발명해 주지 않는다. 어떤 사후 신호가 "좋음"을 대리하는지 정의하고, 그 신호의 편향을 어떤 매칭으로 깎아내야 reward로 쓸 만한지 결정하는 것 — 이 두 가지가 본 논문이 자동화하지 못한 자리이고, 사용자가 검증해야 할 자리다.
