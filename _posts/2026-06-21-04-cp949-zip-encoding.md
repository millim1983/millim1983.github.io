## 0. 어떤 파일은 깨지고 어떤 파일은 멀쩡하다

공공기관 사이트에서 받은 첨부는 대개 ZIP으로 묶여 있다. 그걸 코드로 풀자 한글 파일명이 깨졌다. `사업공고문.hwp`가 `���battibe���.hwp` 같은 글자로 나왔다. 그런데 이상한 건, 같은 묶음 안에서도 어떤 파일은 멀쩡하고 어떤 파일만 깨졌다는 점이다.

"한글이니까 CP949 문제겠지"가 첫 짐작이었다. 절반만 맞았다. 진짜 원인은 ZIP 명세에 정의된 플래그 비트 한 개였고, 그 비트를 보지 않으면 어떤 파일은 고치고 어떤 파일은 도리어 망가뜨린다.

> **"한글이니까 CP949"는 대표적 경우 하나일 뿐이다. 표준이 정의한 분기를 보지 않으면, 고치려는 코드가 멀쩡한 파일을 깬다.**

이 글은 그 플래그 비트 하나를 찾아낸 과정과, AI의 첫 코드가 왜 절반만 맞았는지를 적는다.

## 1. CP949와 UTF-8이 한 묶음에 섞인다

ZIP 파일은 각 항목의 이름을 바이트로 저장한다. 문제는 그 바이트를 어떤 인코딩으로 읽어야 하는지다. 오래된 윈도우 한글 환경에서 만든 ZIP은 파일명을 CP949(한국어 완성형)로 저장한다. 반면 요즘 도구로 만든 ZIP은 UTF-8로 저장한다.

Python의 `zipfile`은 기본적으로 파일명을 CP437(옛 도스 인코딩)로 디코딩하려 든다. UTF-8 플래그가 없는 항목을 만나면 이름을 CP437로 풀어 버려서, 한글 바이트가 깨진 라틴 문자로 나온다. 그래서 "CP949로 다시 읽어라"가 필요한데 — 모든 항목에 그렇게 하면 안 된다. UTF-8로 저장된 항목까지 CP949로 비틀면 그게 도리어 깨진다. 한 묶음에 두 인코딩이 섞일 수 있다는 게 함정의 핵심이다.

## 2. 어떻게 구분하나 — 플래그 비트 0x800

ZIP 명세(APPNOTE 4.4.4, general purpose bit flag)에는 항목마다 "이 파일명은 UTF-8로 저장됐다"를 표시하는 비트가 있다. 11번 비트, 값으로는 `0x800`이다. 이 비트가 켜져 있으면 이름은 UTF-8이고, 꺼져 있으면 그 외(여기서는 CP949)로 봐야 한다.

Python `zipfile`은 각 항목 정보를 `ZipInfo` 객체로 주는데, 거기에 `flag_bits`가 들어 있다. 그러니 항목마다 이 비트를 확인해 분기하면 된다. 같은 묶음에서 어떤 파일은 멀쩡하고 어떤 파일만 깨진 이유가 여기 있었다. UTF-8 플래그가 켜진 항목은 `zipfile`이 알아서 제대로 읽었고, 꺼진 항목만 CP437로 잘못 읽혀 깨진 것이다.

## 3. AI의 첫 코드는 왜 절반만 맞았나

Claude Code에게 "ZIP의 한글 파일명이 깨지니 고쳐라"라고 했을 때, 첫 코드는 이랬다.

```python
# AI의 첫 초안 — 모든 이름을 CP949로 재해석
name = info.filename.encode("cp437").decode("cp949")
```

명세를 모르는 채 "한글 = CP949"라는 대표 케이스만 적용한 코드다. CP949로 저장된 항목에는 맞다. 하지만 UTF-8로 저장된 항목에까지 이 변환을 적용하면, 멀쩡하던 이름이 거꾸로 깨진다. 테스트에서 어떤 파일은 고쳐지고 어떤 파일은 새로 깨지는 걸 보고, 나는 플래그 비트를 확인하라고 명시적으로 지시했다.

이 코드를 보이는 목적은, 도구의 첫 답이 "가장 흔한 한 가지 경우"에 머물기 쉽다는 걸 보이기 위해서다. 고친 코드는 이렇다.

```python
# core/archive.py — 항목마다 UTF-8 플래그를 보고 분기
def _decode_member_name(info):
    if info.flag_bits & 0x800:        # 11번 비트가 켜졌으면 이름은 UTF-8
        return info.filename          # 그대로 둔다
    try:
        # 플래그가 없으면 CP437로 잘못 읽힌 것 → CP949로 되돌린다
        return info.filename.encode("cp437").decode("cp949")
    except (UnicodeDecodeError, UnicodeEncodeError):
        return info.filename          # 변환 실패 시 원본 유지(추가 손상 방지)
```

차이는 한 줄, `if info.flag_bits & 0x800`이다. 이 한 줄이 "모든 한글을 CP949로"라는 짐작을, "표준이 정의한 대로 항목마다 분기"로 바꾼다.

## 4. 덤으로 — zip slip 방어

파일명을 고친 김에 한 가지를 더 짚었다. ZIP 항목 이름에 `../../etc/passwd` 같은 경로가 들어 있으면, 압축을 풀 때 의도한 폴더 밖에 파일이 쓰일 수 있다(zip slip 취약점). Claude Code가 이 방어 코드는 시키지 않아도 넣었다.

```python
# 풀어낸 경로가 목표 폴더 안에 있는지 확인
out_path = (dest_dir / safe_name).resolve()
if not str(out_path).startswith(str(dest_dir.resolve())):
    raise ValueError("경로 이탈 시도 차단")
```

도구가 보안 관행을 자발적으로 넣은 회색 지대다. 나는 이 방어를 받아들이되, 정말 목표 폴더 기준이 맞는지 한 번 확인했다. 도구의 제안을 수용할 때도 그게 내 맥락에 맞는지는 사람이 본다.

## 5. 사람에게 남는 일

ZIP 인코딩 처리에서 코드 자체는 도구가 다 짰다. CP437→CP949 변환도, 플래그 분기도, zip slip 방어도 결국 Claude Code의 손에서 나왔다. 그럴수록 사람의 일은 코드 작성이 아니라 그 코드가 어떤 경우를 놓치는지 보는 데 있다.

도구의 첫 답은 가장 흔한 한 가지 경우를 잘 처리한다. "한글이니까 CP949"는 그 흔한 경우다. 그러나 표준에는 분기가 정의돼 있고, 실제 데이터에는 그 분기의 양쪽이 다 섞여 들어온다. 어떤 파일은 멀쩡하고 어떤 파일만 깨진다는 관찰에서 "그러면 둘을 가르는 기준이 데이터 안에 있겠다"로 가는 것, 그리고 그 기준을 표준에서 찾아 코드에 강제하는 것은 사람이 한다.

이 단계에서 코딩 에이전트가 보편화된 환경의 사람에게 새로 요구되는 능력은, **표준이 정의한 분기를 코드가 따르도록 강제하는 능력**이다. 도구의 첫 답을 "대표 케이스 하나"로 의심하고, 데이터가 그보다 복잡하다는 걸 증거로 짚어내는 일. 깨진 파일명 하나가 그 의심의 출발점이었다.
