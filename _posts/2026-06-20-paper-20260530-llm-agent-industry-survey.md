> **TL;DR** — 2026년 5월 4일 arXiv에 올라온 산업 응용 중심 LLM 에이전트 서베이. 새 모델이나 학습 기법을 제안한 게 아니라, **현재까지 산업에 들어간 LLM 에이전트의 구조와 사례를 한 장에 모은 지도**다. 코딩이 본업이 아닌 실무자가 "이 동네 풍경이 어떻게 생겼나"를 30분 안에 파악하기 좋다.

## 1. 모델 한눈에 — 에이전트는 모델이 아니라 시스템이다

이 논문이 처음 강조하는 건 **LLM 에이전트는 단일 모델이 아니라 다섯 구성요소의 조합**이라는 점이다.

| 구성요소 | 역할 |
|---|---|
| LLM (인지 엔진) | 추론·계획·자연어 인터페이스 |
| 도구 활용 (MCP 등) | 외부 시스템 호출 |
| 메모리 (RAG·벡터 DB) | 장기 기억과 컨텍스트 보강 |
| 환경 감지 (멀티모달) | 텍스트·이미지·표·음성 처리 |
| 안전장치 (Guardrail) | 정책·환각 방지·출력 검증 |

논문은 이 구조를 다음 도식으로 정리한다.

![LLM-Powered AI Agent System 개요](https://arxiv.org/html/2505.16120v2/figures/Agent_genAI.png)

에이전트의 종류도 셋으로 나눈다. ① 소프트웨어 기반(챗봇·자동화 비서·코딩 도우미) ② 물리적(자율주행·로봇·드론) ③ 적응형/하이브리드(IoT·디지털 트윈 등 두 세계를 잇는 형태).

## 2. 데이터 준비 — 서베이라 자체 데이터는 없지만

자체 데이터셋을 만든 논문이 아니다. 대신 **산업이 LLM 에이전트에 어떤 데이터를 먹이고 있는지**를 사례별로 정리한다.

- **고객 서비스**: 회사 내부 문서·과거 상담 로그를 벡터 DB로 색인. RAG로 응답 시점에 검색.
- **소프트웨어 개발**: 코드베이스 전체를 임베딩. 변경 이력·이슈 트래커를 같이 인덱싱하는 시도가 늘었다.
- **금융**: 시계열 수치 + 뉴스 텍스트 + 차트 이미지를 한 에이전트가 동시에 받음. **멀티모달이 단순 옵션이 아니라 거래 신호 생성의 전제 조건**이 됐다는 점이 새롭다.

이 절의 핵심 메시지는 한 줄이다 — **모델이 아니라 데이터 파이프라인이 산업 도입의 병목이다.**

## 3. 학습 절차 — 새 학습 기법은 없다

서베이라서 새 사전학습 방법이나 손실함수 같은 건 다루지 않는다. 다만 산업이 실제로 쓰는 학습 패턴 두 가지를 정리한다.

- **Zero-shot / Few-shot 프롬프팅** — 대다수 산업 도입의 1차 방식. 프롬프트 엔지니어링이 사실상의 "학습"이 된다.
- **도메인 특화 파인튜닝** — 의료·법률·금융처럼 어휘가 일반 코퍼스와 크게 다른 영역에서만 추가 비용을 감수.

저자의 진단은 명확하다 — **2026년 시점 산업 도입의 80 %는 파인튜닝 없이 프롬프트와 RAG로 해결되고 있다.**

## 4. 파인튜닝 — 적응형 에이전트의 자리

이 논문이 파인튜닝을 따로 다루는 자리는 한 군데다 — **적응형/하이브리드 에이전트**. 물리 세계와 인터페이스하는 에이전트(자율주행·로봇)는 도메인 특화 파인튜닝 또는 강화학습 기반 정책 학습이 필수다. 텍스트 챗봇 영역과 달리 "프롬프트만으로 충분"이 통하지 않는다.

이 구분이 본 블로그가 별도 글로 다룬 [피지컬 AI 페타바이트 데이터](/ax/ax-03-physical-ai-petabyte/) 흐름과 정확히 연결된다.

## 5. 성능 평가 — 표준 벤치마크가 아직 없다

이 절이 논문의 솔직한 자리다. 저자는 **"표준화된 벤치마크와 평가 지표의 부재"** 를 LLM 에이전트 산업화의 가장 큰 미해결 문제로 꼽는다.

현재 쓰이는 평가 방향은 셋으로 나뉜다.

| 축 | 무엇을 본다 |
|---|---|
| 작업 중심 | 작업 성공률·완료 시간·재시도 횟수 |
| 대화 중심 | 응답 정확도·일관성·맥락 유지 |
| 인간 중심 | 사용자 만족도 조사·A/B 테스트 |

세 축이 따로 노는 것 자체가 미성숙의 증거다. 한 산업이 다른 산업의 에이전트 성능을 비교할 공통 척도가 없다.

## 6. 우리는 이걸 어떻게 활용하면 좋겠는가 — 실무 함의

본 글의 절반은 여기에 쓴다. 코딩이 본업이 아닌 실무자 입장에서 이 서베이가 던지는 신호 세 가지.

### 6-1. "에이전트 도입"은 모델 선정 문제가 아니다

이 논문의 가장 강한 메시지는 **에이전트 시스템 도입에서 LLM 선택은 다섯 구성요소 중 하나일 뿐**이라는 점이다. 도구 호출·메모리·환경 감지·안전장치가 같이 풀려야 산업에 들어간다. RFP를 쓰거나 사업 제안서를 검토할 때, "Claude를 쓸지 GPT를 쓸지"에 분량이 집중된 제안은 다섯 중 하나에만 답을 가진 셈이다.

> **모델은 사올 수 있다. 다섯 구성요소를 묶는 시스템은 사올 수 없다.**

### 6-2. 80 %는 프롬프트와 RAG로 해결된다는 사실의 두 얼굴

좋은 면: 도입 비용이 작다. 데이터셋을 모아 모델을 학습시키는 큰 단위 작업 없이도 산업 적용이 시작된다.

위험한 면: 그래서 너도나도 챗봇 하나로 "AI 도입 완료"라고 말한다. 본 블로그가 일관되게 다루는 "사용자의 일이 어디에 자리 잡는가" 질문이 여기서도 그대로 살아난다 — **프롬프트와 RAG로 만든 에이전트는 사용자가 결정한 데이터 경계·정책·검증 절차에 그 품질이 정확히 비례한다.** 모델 선택만큼 그 결정의 무게가 크다.

### 6-3. 평가 부재 = 도입 평가표의 자유도

표준 벤치마크가 없다는 사실은, 산업 도입 시점에 **평가표를 발주자가 정의할 수 있다는 뜻**이다. AX 사업 제안서나 RFP에서 평가 항목을 어떻게 짤지가 사실상 사업의 성패를 가른다. 작업 성공률·대화 일관성·인간 만족도 셋 중 어느 축에 가중치를 둘지가 미리 박혀 있어야 한다. 그게 안 되어 있으면 도입 1년차에 "이게 잘 굴러간 게 맞나"라는 질문에 답하지 못한다.

## 7. 한 줄 정리

> **2026년 5월의 LLM 에이전트 풍경 = 다섯 구성요소의 조합 + 멀티모달 + 표준 평가 부재.**

산업 도입을 준비하는 실무자가 새 모델 발표보다 먼저 챙겨야 할 건 자기 조직의 데이터 파이프라인과 평가 정의다.
